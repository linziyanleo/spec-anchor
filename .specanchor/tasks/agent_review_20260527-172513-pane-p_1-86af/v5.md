# Plan v5: spec-anchor 触发率优化——五层触发体系实现审查

## Context / Goals

spec-anchor 作为 Claude Code 插件的触发率远低于 superpowers，根因分析发现 5 个结构性差距：
1. 无 SessionStart hook——无法在 session 启动时自动注入上下文
2. description 过长且描述性（~200 字），skill 匹配分数低
3. 仅有 `~/.claude/skills/` 单入口，缺少 plugin `skills/` 入口
4. 无防跳过对策——agent 倾向跳过"不紧急"的 skill
5. 无 post-coding chain——编码后不回流 findings/sediment

**目标**：实现五层触发体系，使 spec-anchor 在含 `anchor.yaml` 或 `.specanchor/` 的项目中自动激活，覆盖 session 启动→编码前→编码后全链路。

**关键约束**：spec-anchor 必须作为唯一的 spec/规划 skill，不依赖 superpowers。可参考 superpowers 的实现模式，但新代码中不引用 superpowers。

## Non-goals

- 不修改已有的 `references/integrations/superpowers.md` 等兼容性文档
- 不修改 `specanchor-boot.sh` 或其他已有脚本的行为
- 不改变 anchor.yaml 格式或 .specanchor/ 目录结构
- 不替换已有的 CLAUDE.md 注入机制（`specanchor-boot-install.sh`）——两者共存互补

## Proposed approach

以下是**已实现且代码已同步**的五层触发体系，提交审查：

### 层 1：SessionStart Hook 注入

**文件：`hooks/hooks.json`**
```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|clear|compact",
        "hooks": [
          {
            "type": "command",
            "command": "\"${CLAUDE_PLUGIN_ROOT}/hooks/run-hook.cmd\" session-start",
            "async": false
          }
        ]
      }
    ]
  }
}
```

**文件：`hooks/run-hook.cmd`** — 跨平台 polyglot 入口（cmd.exe + bash），从 superpowers 模式复用。cmd 部分查找 Git for Windows 的 bash，Unix 部分直接 exec bash。已设置可执行权限（`-rwxr-xr-x`）。

**文件：`hooks/session-start`** — 条件注入脚本（核心）：
```bash
#!/usr/bin/env bash
set -euo pipefail

# 条件检测：仅在 anchor.yaml 或 .specanchor/ 存在时注入
if [ ! -f "anchor.yaml" ] && [ ! -d ".specanchor" ]; then
  printf '{}\n'
  exit 0
fi

# 从 anchor.yaml 提取项目名，fallback 到目录名
project_name="$(grep -m1 '^project_name:' anchor.yaml ... || basename pwd)"

# escape_for_json() 使用 bash parameter substitution
# 处理 \, ", \n, \r, \t

# 注入内容使用 <IMPORTANT> 标签（非 EXTREMELY_IMPORTANT，避免与其他插件竞争）
# 包含：MANDATORY invoke 指令 + 5 条触发场景（含 spec 管理域）+ 5 条防跳过 + 条件性 Post-Coding Chain
# Post-Coding Chain 使用 specanchor-finding.sh 脚本接口而非直接写模板

# 三平台输出：
# Cursor → additional_context (snake_case)
# Claude Code → hookSpecificOutput.additionalContext (nested)
# Copilot CLI / 其他 → additionalContext (top-level)
```

### 层 2：命令式短 description

**根 SKILL.md frontmatter** 从 ~200 字描述性文本改为：
```yaml
description: "MUST invoke in projects with anchor.yaml or .specanchor/ — before code changes, spec/context management, alignment checks, handoff, findings, or sediment work. Loads coding standards, module contracts, and active tasks."
```

关键变化：以 `MUST invoke` 开头（命令式），~180 字符，包含触发条件（`anchor.yaml or .specanchor/`）、编码触发（`before code changes`）和非编码触发域（`spec/context management, alignment checks, handoff, findings, sediment`），保持关键 SpecAnchor 命令的可发现性。

### 层 3：宽触发面积（双入口）

**新建 `skills/spec-anchor/SKILL.md`** — 插件 skill 发现入口（thin wrapper）：
```markdown
---
name: spec-anchor
description: "MUST invoke in projects with anchor.yaml or .specanchor/ — before code changes, spec/context management, alignment checks, handoff, findings, or sediment work. Loads coding standards, module contracts, and active tasks."
---

<HARD-GATE>
If this project has `anchor.yaml` or `.specanchor/`, load SpecAnchor context before code changes.
Specs contain constraints not visible in code alone — even for small edits, assemble context first.
</HARD-GATE>

The full spec-anchor skill definition is in the plugin root `SKILL.md`. Read and follow `../../SKILL.md` for all commands, protocols, and references.
```

结合已有的 `~/.claude/skills/spec-anchor/SKILL.md`（standalone 入口），形成双入口覆盖。

**新建 `.claude-plugin/plugin.json`** + **`.claude-plugin/marketplace.json`** — 插件 manifest，使 Claude Code 发现 `hooks/` 和 `skills/`。

### 层 4：防跳过对策

三重防线：
1. **SessionStart hook 注入**：`<IMPORTANT>` 标签包裹 5 条"Do NOT skip because"反论
2. **HARD-GATE 块**：在 SKILL.md 和 skills/spec-anchor/SKILL.md 中，引导在 anchored 项目中加载 context（不强制具体工作流）
3. **命令式 description**：`MUST invoke` 语义暗示非可选

5 条反跳过条目：
- "This is just a simple change" → 小改也要遵守 project contracts
- "I already know the codebase" → spec-anchor 加载活跃任务和最新 findings
- "I'll check later" → context BEFORE coding 防止 misalignment
- "The user didn't ask for it" → 项目级要求，非可选
- "I can infer conventions from code" → specs 含代码中不可见的约束

### 层 5：链式调用闭环

**根 SKILL.md 新增 Post-Coding Chain 段**：
```
After completing a coding task, run through this chain:
1. Alignment Check — compare changes against loaded specs (specanchor_check) [always]
2. Record Findings — if new facts, contradictions, risks, or drift were discovered,
   record via: SPECANCHOR_SKILL_DIR="$SA_SKILL_DIR" bash "$SA_SKILL_DIR/scripts/specanchor-finding.sh" new --topic=<slug> --summary=<line>
3. Evaluate Sediment — if findings reveal spec drift, draft a Sediment Proposal

Always run Alignment Check. Record Findings and Evaluate Sediment can be skipped when no new facts, risks, or drift were discovered.
```

**SessionStart hook 注入中也包含 Post-Coding Chain**：使用自然语言引导（"use the spec-anchor finding command"），不硬编码 bare path——agent 在 boot 后已通过 `$SA_SKILL_DIR` 知道脚本位置。Alignment Check 始终执行；Record Findings 和 Evaluate Sediment 仅在发现新事实或 drift 时执行。

## Risks & open questions

1. **Bash 3.2 兼容性**：`escape_for_json()` 使用 `${s//pattern/replacement}` 和 `$'\n'`——在 Bash 3.2（macOS 默认）中支持，但 `$'\n'` 在 pattern 位置的行为在极早期 bash 版本中可能不同。已在 macOS 实际测试通过。

2. **双入口冲突**：standalone 安装（`~/.claude/skills/spec-anchor/`）和 plugin 安装（`skills/spec-anchor/`）同时存在时，Claude Code 是否会重复加载？预期：Claude Code 按 name 去重，两个同名 skill 只加载一个。

3. **`<IMPORTANT>` vs `<EXTREMELY_IMPORTANT>`**：选择 `<IMPORTANT>` 而非 `<EXTREMELY_IMPORTANT>` 是为了避免与 superpowers 等其他插件的注入竞争。如果用户不安装 superpowers，可以升级为 `<EXTREMELY_IMPORTANT>`。

4. **project_name 提取**：使用 `grep -m1 '^project_name:'` 做简单 YAML 解析——不处理引号嵌套、多行值等边界情况。Fallback 到 `basename $(pwd)` 足够安全。

5. **hook async: false**：同步执行意味着 session-start 脚本必须快速完成。当前实现仅做文件存在性检查 + 字符串拼接，耗时 < 100ms，无性能风险。

6. **Plugin Packaging surface contract 后续跟进**（owner=方壶）：当前 hooks/, skills/, .claude-plugin/ 无独立 module spec。这些目录是 packaging 层，不含业务逻辑，其契约由 Claude Code 插件规范外部定义。完整的 Plugin Packaging module spec 作为独立任务在触发率优化合并后跟进。
