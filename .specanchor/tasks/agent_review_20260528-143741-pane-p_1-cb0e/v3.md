# Plan v3: Guarantee spec-anchor activation at conversation start

## Context / Goals

**Problem**: spec-anchor's SessionStart hook fires and injects an `<IMPORTANT>` block telling agents to invoke the spec-anchor skill. However, in practice the trigger rate is ~70-80% — agents often skip it for "simple" questions or when no obvious coding task is detected in the first user message.

**Root cause**: The hook only injects *advice* ("please invoke spec-anchor"), not *substance*. Whether the agent actually calls the Skill tool depends on its own judgment. Different agents (Claude Code, Cursor, Copilot) have different compliance rates with advisory instructions.

**Team context**: The team uses heterogeneous Coding Agents but wants unified Spec management via spec-anchor. The constraint `anchor.yaml || .specanchor/` in the hook is correct (projects that use spec-anchor will always have these). The gap is between "hook fires" and "agent actually loads spec context".

**Goals**:
1. Ensure that in 95%+ of conversations, the agent has spec-anchor constraints loaded before any code change — regardless of which agent is used.
2. Do NOT require changes to team members' personal agent configurations.
3. Maintain the current plugin-based distribution model (install via `/plugin marketplace add`).

## Non-goals

- Forcing spec-anchor on projects that don't have `anchor.yaml` / `.specanchor/` (current gate is correct).
- Replacing the skill invocation entirely (the skill provides interactive capabilities beyond just context loading).
- Supporting agents that don't support any form of session hooks or system prompts.
- Changing the trigger scope beyond code edits, reviews, and spec/context work.

## Proposed approach

### Layer 1: Hook inlines critical spec summary (guarantees baseline)

Change the `session-start` hook to **inline** a compact project context summary directly in the hook output. This ensures even if the agent never calls the Skill tool, it already has:
- Global coding standards (top constraints)
- Active task list (what's in progress)
- Module contracts summary (key interfaces)

Implementation (complete error-safe pattern):
```bash
# In hooks/session-start, after the existing IMPORTANT block:
boot_err_file="$(mktemp -t sa-hook-boot.XXXXXX)"
trap 'rm -f "$boot_err_file"' EXIT

spec_summary=""
spec_status="ok"
if boot_output="$("$PLUGIN_ROOT/scripts/specanchor-boot.sh" --format=inline-brief 2>"$boot_err_file")"; then
  spec_summary="$boot_output"
else
  spec_status="error"
  spec_summary="Boot failed: $(head -c 200 "$boot_err_file")"
fi

if [ "$spec_status" = "ok" ] && [ -n "$spec_summary" ]; then
  session_context="${session_context}\n\n<spec-anchor-context>\n${spec_summary}\n</spec-anchor-context>"
elif [ "$spec_status" = "error" ]; then
  session_context="${session_context}\n\n<spec-anchor-context status=\"error\">\n${spec_summary}\n</spec-anchor-context>"
fi
```

Key properties:
- `mktemp` creates the temp file; `trap EXIT` ensures cleanup
- Boot failure is captured in `$boot_err_file`, hook always exits 0 with valid JSON
- Agent sees `status="error"` and knows context is missing — never silently degrades
- Under `set -euo pipefail`: the `if` construct handles the non-zero exit from boot without triggering errexit

Budget: Configurable via `anchor.yaml` field `hook.inline_budget_tokens`, default 600, bounds [200, 1500], validated at boot time (warning on out-of-bounds, clamp to nearest bound).

### Layer 2: Strengthen skill description (bounded scope)

Change the skill description from current:
```
"MUST invoke in projects with anchor.yaml or .specanchor/ — before code changes..."
```
To:
```
"MUST invoke in projects with anchor.yaml or .specanchor/ — before code changes, spec/context management, reviews, or process skills. Loads coding standards, module contracts, and active tasks."
```

Key change: explicit enumeration of trigger scenarios. Keep `<IMPORTANT>` wrapper (not escalate to `<EXTREMELY_IMPORTANT>`). The inline spec summary (Layer 1) provides passive baseline for non-coding prompts without requiring skill invocation.

### Layer 3: Enhance existing `specanchor-boot-install.sh` with strengthened trigger rule

**No new command or script.** Enhance the existing `specanchor-boot-install.sh` / `specanchor-init.sh --install-boot=<targets>` to inject a strengthened trigger rule into the target agent config files:

For CLAUDE.md:
```markdown
## Spec Anchor (mandatory)
Invoke the spec-anchor skill before code changes, spec/context management, reviews, or process skills.
```

For .cursorrules / AGENTS.md: equivalent format per platform.

This uses the existing command routing and installation flow — no new `sa init` command, no new script.

## Risks & open questions

1. **Token budget**: Inlining spec summary in every session start adds ~600 tokens (configurable). For large projects, priority-based inclusion prevents bloat. The budget is hard-capped at 1500 tokens.

2. **Stale inline context**: If specs change mid-session, the inlined summary from session start becomes stale. Mitigation: The skill invocation still provides fresh context when invoked; the inline is a baseline, not a replacement.

3. **Boot failure visibility**: On failure, `<spec-anchor-context status="error">` makes the gap visible. The hook itself always exits 0 (valid JSON) to avoid breaking session start. The `if` construct ensures `set -e` doesn't kill the hook.

4. **Superpowers interaction**: When both superpowers and spec-anchor are installed, `using-superpowers` scans skills and may invoke spec-anchor. The bounded description (Layer 2) reduces unnecessary activation on non-coding prompts while maintaining reliable activation for code work.
