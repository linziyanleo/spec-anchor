# Plan v1: Guarantee spec-anchor activation at conversation start

## Context / Goals

**Problem**: spec-anchor's SessionStart hook already fires and injects an `<IMPORTANT>` block telling agents to invoke the spec-anchor skill. However, in practice the trigger rate is ~70-80% — agents often skip it for "simple" questions or when no obvious coding task is detected in the first user message.

**Root cause**: The hook only injects *advice* ("please invoke spec-anchor"), not *substance*. Whether the agent actually calls the Skill tool depends on its own judgment. Different agents (Claude Code, Cursor, Copilot) have different compliance rates with advisory instructions.

**Team context**: The team uses heterogeneous Coding Agents but wants unified Spec management via spec-anchor. The constraint `anchor.yaml || .specanchor/` in the hook is correct (projects that use spec-anchor will always have these). The gap is between "hook fires" and "agent actually loads spec context".

**Goals**:
1. Ensure that in 95%+ of conversations, the agent has spec-anchor constraints loaded before any code change — regardless of which agent is used.
2. Do NOT require changes to team members' personal agent configurations.
3. Maintain the current plugin-based distribution model (install via `/plugin marketplace add`).

## Non-goals

- Forcing spec-anchor on projects that don't have `anchor.yaml` / `.specanchor/` (current gate is correct).
- Replacing the skill invocation entirely (the skill provides interactive capabilities beyond just context loading).
- Supporting agents that don't support any form of session hooks or system prompts.

## Proposed approach

### Layer 1: Hook inlines critical spec summary (guarantees baseline)

Change the `session-start` hook to **inline** a compact project context summary directly in the hook output, rather than just advising "invoke the skill". This ensures even if the agent never calls the Skill tool, it already has:
- Global coding standards (top constraints)
- Active task list (what's in progress)
- Module contracts summary (key interfaces)

Implementation:
```bash
# In hooks/session-start:
# After the existing IMPORTANT block, append a condensed spec summary
spec_summary="$("$PLUGIN_ROOT/scripts/specanchor-boot.sh" --format=inline-brief 2>/dev/null || true)"
if [ -n "$spec_summary" ]; then
  session_context="${session_context}\n\n<spec-anchor-context>\n${spec_summary}\n</spec-anchor-context>"
fi
```

Budget: ~500-800 tokens for the inline summary (configurable via `anchor.yaml` field `hook_budget`).

### Layer 2: Strengthen skill description for "session start" semantics

Change the skill description from current:
```
"MUST invoke in projects with anchor.yaml or .specanchor/ — before code changes..."
```
To:
```
"MUST invoke at the START of every conversation in projects with anchor.yaml or .specanchor/ — loads project constraints, coding standards, and active tasks before any response or code change"
```

Key change: adding "at the START of every conversation" and "before any response" — this matches the trigger pattern of `using-superpowers` which achieves near-100% trigger rate.

### Layer 3: Provide `sa init` command that writes to project CLAUDE.md

Add a one-time setup command that appends a hard rule to the project's `.claude/CLAUDE.md` (or equivalent for other agents):

```markdown
## Spec Anchor (mandatory)
Always invoke the spec-anchor skill at conversation start. Do not skip for any reason.
```

This is the **highest priority** instruction layer for Claude Code. For Cursor it goes in `.cursorrules`, for Copilot in `AGENTS.md`.

The `sa init` command detects which agent config files exist and writes to all of them.

### Layer 4 (belt-and-suspenders): Hook output includes skill-invoke instruction in `<EXTREMELY_IMPORTANT>` wrapper

Superpowers uses `<EXTREMELY_IMPORTANT>` tags. Adopt the same pattern:

```bash
session_context="<EXTREMELY_IMPORTANT>
This project uses spec-anchor. You MUST invoke the spec-anchor skill before any response.
The following project constraints are already loaded for reference, but invoke the skill for full interactive capabilities.
</EXTREMELY_IMPORTANT>

<spec-anchor-context>
${spec_summary}
</spec-anchor-context>"
```

## Affected files

| File | Change |
|------|--------|
| `hooks/session-start` | Add inline spec summary + strengthen wrapper tags |
| `skills/spec-anchor/SKILL.md` | Update description field |
| `scripts/specanchor-boot.sh` | Add `--format=inline-brief` output mode (~500 token budget) |
| `anchor.yaml` (schema) | Optional `hook_budget` field |
| New: `scripts/specanchor-init-agent-config.sh` | Write rules to CLAUDE.md/.cursorrules/AGENTS.md |
| `SKILL.md` (root) | Document the `sa init` command |

## Risks & open questions

1. **Token budget**: Inlining spec summary in every session start adds ~500-800 tokens to every conversation. For large projects with many specs, this could bloat. Mitigation: strict budget cap + priority-based inclusion.

2. **Stale inline context**: If specs change mid-session, the inlined summary from session start becomes stale. Mitigation: The skill invocation (Layer 2) still provides fresh context when invoked; the inline is a baseline, not a replacement.

3. **`specanchor-boot.sh --format=inline-brief` failure**: If the boot script fails (e.g., malformed anchor.yaml), the hook should still succeed with the advisory-only fallback. Current design has `|| true` but we should ensure graceful degradation.

4. **Multi-agent config fragmentation**: `sa init` writing to CLAUDE.md + .cursorrules + AGENTS.md means maintaining 3 formats. Risk of drift. Mitigation: generate from single template, version-stamp the output.

5. **Superpowers interaction**: When both superpowers and spec-anchor are installed, `using-superpowers` already scans skills and may invoke spec-anchor. The strengthened description (Layer 2) should make this more reliable, but we need to avoid duplicate context injection.

## Verification plan

1. **Manual test (Claude Code)**: Start a new session in a project with `anchor.yaml`. Verify that:
   - The hook output contains `<spec-anchor-context>` with actual spec content
   - The agent invokes spec-anchor skill before responding to the first message
   - Even with message "what time is it?" (non-coding), baseline context is present

2. **Manual test (Cursor)**: Same project, verify `.cursorrules` injection + hook output works

3. **Token budget check**: Measure actual token count of inline summary for the spec-anchor project itself (dogfood). Should be < 800 tokens.

4. **Failure mode test**: Corrupt `anchor.yaml`, verify hook still outputs advisory-only fallback without crashing.

5. **Idempotency test**: Run `sa init` twice, verify no duplicate rules in CLAUDE.md.
