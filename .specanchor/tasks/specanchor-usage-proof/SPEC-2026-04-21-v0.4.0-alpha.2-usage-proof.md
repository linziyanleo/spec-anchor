# SpecAnchor v0.4.0-alpha.2 — Usage Proof

## 0. Agent Execution Prompt

You are implementing SpecAnchor `v0.4.0-alpha.2: Usage Proof`.

Your goal is to prove that an external user can install SpecAnchor into realistic clean projects and run the governance workflow successfully.

Do not expand product scope. Do not redesign the core protocol. Do not add RAG, vector search, MCP server, daemon mode, or UI.

# 1. Product Goal

`v0.4.0-alpha.1` proved the repository can self-boot, run shell tests, and expose a trustable core.

`v0.4.0-alpha.2` must prove usage.

The release should answer:

> Can a new user install SpecAnchor into a clean project, initialize it, load specs, resolve anchors, validate health, and understand what happened?

The answer must be demonstrated by committed examples, tests, and docs.

## 4.2 Example 1: `minimal-full-project`

Purpose:

Prove SpecAnchor can initialize a clean project in `full` mode.

Initial committed files:

```text
examples/minimal-full-project/
├── README.md
├── src/auth/login.md
```

Do not commit generated `.specanchor/` or `anchor.yaml` inside this example unless the test explicitly needs a fixture. The smoke test should copy this example into a temp directory, install the skill, run init, then verify generated files.

Smoke flow:

```bash
tmp="$(mktemp -d)"
cp -R examples/minimal-full-project "$tmp/project"

skill_dir="$PWD"

  bash "$project_dir/.cursor/skills/specanchor/scripts/specanchor-init.sh" \

  bash "$project_dir/.cursor/skills/specanchor/scripts/specanchor-boot.sh" \

  --strict
```

## 4.3 Example 2: `parasitic-openspec-project`

Purpose:

Prove SpecAnchor can govern an existing spec directory without moving files.

```text
examples/parasitic-openspec-project/
├── README.md
├── specs/auth.md
├── specs/billing.md
├── src/auth/login.md
├── src/billing/invoice.md
└── expected/README.md
```

The test should copy this example into a temp directory, create or commit a minimal `anchor.yaml` in parasitic mode, and run boot / doctor / resolve.

Recommended `anchor.yaml` for the fixture:

```yaml
specanchor:
  version: "0.4.0-alpha.2"
  project_name: "parasitic-openspec-project"
  mode: "parasitic"

  paths:
    global_specs: ".specanchor/global/"
    module_specs: ".specanchor/modules/"
    task_specs: ".specanchor/tasks/"
    archive: ".specanchor/archive/"
    module_index: ".specanchor/module-index.md"
    project_codemap: ".specanchor/project-codemap.md"

  sources:
    - name: "openspec-fixture"
      type: "openspec"
      path: "specs/"
      description: "Existing external specs used by the parasitic example."

  writing_protocol:
    schema: "openspec-compat"
    schema_recommend: true

  coverage:
    scan_paths:
      - "src/**"
      - "specs/**"
    ignore_paths: []
    check:
      stale_days: 14
      outdated_days: 30
      warn_recent_commits_days: 14
```

Smoke flow:


# 13. Design Guidance

## 13.1 Why Examples First

The product already has core scripts and self-governance. The next risk is not lack of features; it is lack of user proof.

Examples turn claims into runnable evidence.

## 13.2 Keep the Index Light

SpecAnchor should follow the same general architecture principle as memory systems that use a lightweight always-loaded index plus detailed files loaded on demand.

For SpecAnchor, this means:

* README points to examples.
* `docs/USAGE_PROOF.md` explains the flow.
* examples contain concrete project fixtures.
* tests prove the examples work.
* core scripts remain deterministic.

Do not move all example details into `SKILL.md`.

## 13.3 Deterministic First

`specanchor-resolve.sh` should remain deterministic-first:

* file path match
* source path match
* module index match
* known spec path match
* optional simple keyword fallback

Do not make alpha.2 depend on LLM semantic matching.
