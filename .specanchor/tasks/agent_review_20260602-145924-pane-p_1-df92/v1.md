# Plan v1: Meta-Harness 协议提升——failure classification + contract compilation awareness + verification mode

## Context / Goals

论文 "Meta-Engineering Harnesses for AI-Native Software Production" (arXiv:2605.25665v1, May 2026) 描述了一个 contract-driven adversarial verification 架构，经 Claude + Codex 联合分析，提炼出三项与 SpecAnchor 直接相关的协议改进。

**目标**：在不改动脚本的前提下，将以下概念引入 SpecAnchor 协议文档和模板：

1. **Finding failure_class** — 在 Finding frontmatter 中增加失败分类字段，建立 failure → action 的映射
2. **Task Spec 两遍 contract 意识** — 在 sdd-riper-one 模板的 §4 Plan 阶段增加 Contract Compilation Checklist
3. **Review Verdict verification_mode** — 在 sdd-riper-one 模板的 §6 增加验证模式元数据记录

附带改动：dual-agent-review 的 codex-review-v1.md prompt 增加 spec completeness review section。

**约束**：
- 本轮只改协议/模板文档，不动脚本（脚本承接是后续任务）
- spec-anchor 是 contract/context/calibration control plane，不是执行引擎
- 新字段必须是可选的（null 默认），保证向后兼容

## Non-goals

- 不修改 `specanchor-finding.sh`、`specanchor-validate.sh` 或任何 `.sh` 脚本
- 不实现 metrics 汇总脚本
- 不实现 specialization registry
- 不为 dual-agent-review 增加 independence mode（已明确放弃）
- 不改动现有 Finding 的 `type` 字段语义

## Proposed approach

### Step 1: Finding failure_class 字段

**文件**: `references/concepts/findings-ledger.md`

在 §3 Frontmatter Schema 中增加 `failure_class` 可选字段定义：

```yaml
failure_class: null  # null | bug | spec_gap | noise | contract_ambiguity
```

增加 §3.x "Failure Class 字段" 说明表：

| failure_class | 含义 | 推荐动作 |
|---|---|---|
| `bug` | 实现违反了明确的 spec/contract 条款 | 修 implementation；promote regression test |
| `spec_gap` | spec/contract 遗漏了必要行为 | sediment proposal → 补 contract/template |
| `noise` | 环境 / CI / 工具链无关失败 | 校准 verifier/CI 配置；可标 visibility=hidden |
| `contract_ambiguity` | spec/contract 允许多种有效解释 | refine contract；不要重试 implementation |
| `null` | 不适用或未分类 | 兼容旧 finding |

与现有 `type` 字段的关系：
- `type` 描述发现的形态（fact/contradiction/risk/...）
- `failure_class` 描述失败的来源和处置路径
- 两者正交，不冲突

路由影响（advisory，不自动执行）：
- `spec_gap` → sediment pipeline 优先候选
- `bug` → 不走 sediment，留在 task scope
- `noise` → 建议 visibility=hidden
- `contract_ambiguity` → 先 refine contract 再重跑

**文件**: `references/templates/finding-template.md`

frontmatter 中增加 `failure_class: null` 行，填写指引段增加说明。

### Step 2: Task Spec 两遍 Contract Awareness

**文件**: `references/schemas/sdd-riper-one/template.md`

在现有 §4.1 File Changes 之前插入 §4.0 Contract Compilation Checklist：

```markdown
### 4.0 Contract Compilation Checklist

> 两遍 contract 编译意识。仅在 Plan 阶段填写；不需要全部满足，但空项需标注"N/A + 原因"。

#### Pass 1: Completeness（是否遗漏了关键约束？）
- [ ] API/UI surface 已列出
- [ ] 状态转移已定义（如有状态机）
- [ ] Invariants 已列出
- [ ] Business rules 已编码（不只是"正确实现"）
- [ ] Auth/data dependencies 已声明
- [ ] Error taxonomy 已列出（不只是 try/catch）
- [ ] Acceptance criteria 可验证

#### Pass 2: Scope & Ambiguity（是否过度指定或歧义？）
- [ ] Out-of-scope 已显式标注
- [ ] 不存在"可被多种方式解读"的条款（或已消歧）
- [ ] 不存在 unsupported requirements（标了但实际无法实现的）
```

## Risks & open questions

1. **failure_class 与 type 的正交性**：两者是否真的不冲突？例如 `type=contradiction` + `failure_class=spec_gap` 是否自然？→ 认为是的：contradiction 描述形态（两处说法不一致），spec_gap 描述来源（spec 漏了东西导致矛盾）。
2. **§4.0 的采用成本**：10 项 checklist 是否过重？→ 设计为建议性（"不需要全部满足，但空项需标注 N/A"），不是 gate。
3. **verification_mode 的实际填写率**：agent 是否会认真填？→ 作为 Review Verdict 的一部分，已在 review phase 自然出现；且是 advisory 记录，不做自动校验（校验在第二步脚本任务中）。
4. **dual-agent-review prompt 改动是否影响 Codex 输出 schema**：→ 不影响，spec completeness 问题使用现有 `category: unclear-requirements`，不引入新 schema 字段。

## Verification plan

1. **语法完整性**：改动后的 markdown 文件无格式错误、frontmatter YAML 合法
2. **向后兼容**：现有 Finding 文件（`failure_class` 为 null 或缺失）不被新协议判定为非法
3. **模板一致性**：sdd-riper-one 模板章节编号连续、无重复
4. **`specanchor-validate.sh` 不报新错**：运行现有 validate 脚本确认不因新字段报错（脚本尚不识别新字段，应 pass through）
5. **Module Spec 同步**：`references.spec.md` 更新以反映新增协议内容
