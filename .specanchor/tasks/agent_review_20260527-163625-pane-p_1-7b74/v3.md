# Plan v3: SpecAnchor 触发率优化——五层触发体系实现

## Context / Goals

**问题**：spec-anchor 在 Claude Code 技能列表中的触发率远低于 superpowers 插件。通过对 superpowers v5.0.7 的逆向分析，识别出 5 个结构性差距：SessionStart hook 注入、命令式短 description、宽触发面积、防跳过对策、链式调用。

**约束**：
- spec-anchor 当前是 `~/.claude/skills/` 下的独立 skill，不是 Claude Code plugin
- 已有 `specanchor-boot-install.sh` 在 CLAUDE.md 中注入触发块（fallback 层）
- 已有 `spec-anchor-prelude` skill 与 superpowers 集成（但未被系统性连接）
- 必须向后兼容非插件安装方式
- **根目录 SKILL.md 不能移动**——它服务于独立 skill 安装（`~/.claude/skills/`）；plugin 安装通过 `skills/spec-anchor/SKILL.md` thin wrapper 做 skill discovery

**目标**：
1. 将 spec-anchor 从被动 skill 升级为 plugin+skill 混合体
2. 通过 SessionStart hook 在 SpecAnchor 项目中自动注入 additionalContext
3. 重写 description 为命令式短文本
4. 内联防跳过对策到 SessionStart 注入内容
5. 与 superpowers 形成链式调用闭环

## Non-goals

- 不替代 superpowers 的工作流（brainstorm→plan→execute），而是在其之前加载 spec context
- 不在非 SpecAnchor 项目中注入任何内容（条件触发，非无条件）
- 不修改 superpowers 的代码（只通过 spec-anchor 侧的 hook 和 skill 实现集成）
- 不上架 marketplace（第一期，先验证机制有效性）
- 不修改现有 scripts/ 和 references/ 的核心逻辑
- **不移动 SKILL.md**——根目录位置是独立 skill 安装的 contract；plugin skill discovery 通过 `skills/spec-anchor/SKILL.md` thin wrapper 实现

## Proposed approach

### 第一层：SessionStart Hook 注入

**1.1 创建 `.claude-plugin/plugin.json`**——Claude Code 插件发现的必需文件。

```json
{
  "name": "spec-anchor",
  "description": "Spec-anchored development — compile bounded, auditable engineering context for coding agents",
  "version": "0.7.0",
  "author": {
    "name": "方壶"
  },
  "repository": "https://github.com/<org>/spec-anchor",
  "license": "MIT",
  "keywords": [
    "spec",
    "context",
    "coding-standards",
    "module-contracts",
    "spec-anchored-development"
  ]
}
```

**1.2 创建 `.claude-plugin/marketplace.json`**——marketplace 元数据（用于后续分发）。

```json
{
  "name": "spec-anchor",
  "description": "Spec-anchored development context compiler",
  "owner": {
    "name": "方壶"
  },
  "plugins": [
    {
      "name": "spec-anchor",
      "description": "Spec-anchored development — compile bounded, auditable engineering context for coding agents",
      "version": "0.7.0",
      "source": "./"
    }
  ]
}
```

**1.3 创建 `package.json`**——最小化插件元数据，与 `.claude-plugin/plugin.json` 配合。

```json
{
  "name": "spec-anchor",
  "version": "0.7.0",
  "type": "module",
  "description": "Spec-anchored development — compile bounded, auditable engineering context for coding agents"
}
```

**1.4 创建 `hooks/hooks.json`**——注册 SessionStart hook。

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|clear|compact",
        "hooks": [
          {
            "type": "command",
            "command": "\"${CLAUDE_PLUGIN_ROOT}/hooks/run-hook.cmd\" session-start",
            "async": false
          }
        ]
      }
    ]
  }
}
```

**1.5 创建 `hooks/session-start`**——条件注入脚本：
- 检测 cwd 是否有 `anchor.yaml` 或 `.specanchor/`
- 不是 SpecAnchor 项目 → `printf '{}\n'` 静默退出（开销 <1ms，两个文件系统测试）
- 是 SpecAnchor 项目 → 注入 `<IMPORTANT>` 标签的 additionalContext，包含：
  - 项目名（从 anchor.yaml 提取 project_name，fallback "this project"）
  - MANDATORY 指令（编码前必须调用 spec-anchor）
  - superpowers 链式要求（先加载 spec 再 brainstorm/plan/execute）
  - 5 条防跳过表（内联在注入内容中）
- 适配三平台输出格式（Cursor: `additional_context` / Claude Code: `hookSpecificOutput.additionalContext` / Copilot CLI: `additionalContext`）
- Bash 3.2 兼容（macOS 默认 bash 版本）：不使用 `readarray`、`declare -A`、`${var,,}` 等 bash 4+ 特性

**1.6 创建 `hooks/run-hook.cmd`**——跨平台 polyglot 入口（cmd.exe batch + bash），复用 superpowers 的 polyglot 结构。

**1.7 创建 `skills/spec-anchor/SKILL.md`**——plugin skill discovery entry（thin wrapper）。

Claude Code 的 plugin skill discovery 扫描 `skills/*/SKILL.md`（经 superpowers 结构验证：superpowers 无根目录 SKILL.md，所有 skill 在 `skills/brainstorming/SKILL.md`、`skills/systematic-debugging/SKILL.md` 等）。因此需要一个 plugin-local skill entry。

```markdown
---
name: spec-anchor
description: >-
  MUST invoke before writing code in projects with anchor.yaml or .specanchor/ —
  loads coding standards, module contracts, and active tasks to anchor implementation.
---

# SpecAnchor

This is the plugin skill entry. The full skill content is at the plugin root.

Read and follow: `../../SKILL.md`

The plugin root SKILL.md contains boot instructions, command routing, workflow
selection, and all reference pointers. After reading it, run the boot script
it specifies.
```

**设计决策**：
- thin wrapper 而非内容复制——避免两份 SKILL.md 同步维护
- 根目录 SKILL.md 保留完整内容——独立安装（`~/.claude/skills/spec-anchor/`）时直接被 Claude Code 发现
- plugin 安装时通过 `skills/spec-anchor/SKILL.md` 被 plugin discovery 发现，agent 随后读取 `../../SKILL.md` 获取完整内容
- skill identifier：plugin 安装时为 `spec-anchor:spec-anchor`；独立安装时为 `spec-anchor`；SessionStart hook 注入文本使用 "invoke the `spec-anchor` skill" 覆盖两种情况

**关键决策**：
- 用 `<IMPORTANT>` 而非 `<EXTREMELY_IMPORTANT>` 标签——避免与 superpowers 的元技能调度器竞争优先级；两者互补不冲突（superpowers 管「哪些技能要调用」，spec-anchor 管「编码前先加载什么 context」）
- 条件注入（检测 anchor.yaml / .specanchor/）而非无条件注入——spec-anchor 是项目级工具，非通用技能
- 注入内容不包含 spec landscape 全文——那是 skill 被调用后 boot 脚本做的事；hook 只负责「提醒 agent 去调用 skill」

### 第二层：命令式短 Description

重写 SKILL.md 的 frontmatter description：

**当前**（~200 字，解释性）：
```
description: "Context Construction System for AI coding agents——为 agent 编译有边界、可审计、可沉淀的工程上下文..."
```

**目标**（~40 字，命令式）：
```yaml
description: >-
  MUST invoke before writing code in projects with anchor.yaml or .specanchor/ —
  loads coding standards, module contracts, and active tasks to anchor implementation.
```

设计原则：
1. 以 MUST 开头，命令式语气
2. 前置触发条件（"before writing code in projects with..."）
3. 说明做什么（"loads coding standards, module contracts..."）
4. 说明为什么（"to anchor implementation"）
5. 不解释内部概念（不提 Context Bundle, Sediment, Finding 等）

### 第三层：宽触发面积

SessionStart hook 已覆盖项目级自动触发——只要项目有 anchor.yaml / .specanchor/，每次会话都会注入提醒。Description 不再需要列举中英文关键词触发列表，只需覆盖核心场景（"before writing code"）。两层协同实现触发面积最大化：
- Hook 层：项目级自动触发（无需用户消息中包含关键词）
- Description 层：技能列表中的判断依据（命令式，场景明确）

### 第四层：防跳过对策

**4.1 SessionStart 注入中的内联防跳过表**（5 条）：

```
DO NOT RATIONALIZE SKIPPING:
| "The task is too simple" → Simple tasks still must follow project conventions loaded by spec-anchor. |
| "I already know the codebase" → Conventions change. Load current specs. |
| "I'll check specs after coding" → Late-bound spec check produces re-work. Load before coding. |
| "Let me explore first" → spec-anchor tells you WHERE to look. Invoke it first. |
| "This is just a quick fix" → Quick fixes in the wrong pattern create tech debt. Load specs. |
```

**4.2 SKILL.md 中的 `<HARD-GATE>` 区块**：
```markdown
<HARD-GATE>
This skill is MANDATORY in SpecAnchor projects. Do NOT skip boot.
Do NOT write code before the Spec Landscape is loaded.
Do NOT propose architectural decisions without referencing loaded specs.
</HARD-GATE>
```

### 第五层：链式调用

**5.1 SessionStart 注入中的链式引导**：
注入内容包含明确指令：
> "Before invoking any superpowers process skill (brainstorming, writing-plans, executing-plans, systematic-debugging), you MUST first invoke spec-anchor (or spec-anchor-prelude) to load spec context."

**5.2 spec-anchor-prelude description 优化**：
```yaml
description: >-
  MUST invoke BEFORE any superpowers process skill (brainstorming, writing-plans,
  executing-plans) in SpecAnchor projects — loads Spec Landscape first.
```

**5.3 Post-Coding Chain**（在 SKILL.md 中添加）：
编码完成后的 chain-back 步骤：
1. 运行 `specanchor_check` 进行对齐检测
2. 如果有新发现 → 写入 `.specanchor/findings/`
3. 评估是否需要 Sediment Proposal

闭环：
```
SessionStart hook → spec-anchor (boot) → superpowers (brainstorm/plan/execute)
→ spec-anchor (alignment check) → finding/sediment
```
