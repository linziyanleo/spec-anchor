# Plan v1: SpecAnchor 触发率优化——五层触发体系实现

## Context / Goals

**问题**：spec-anchor 在 Claude Code 技能列表中的触发率远低于 superpowers 插件。通过对 superpowers v5.0.7 的逆向分析，识别出 5 个结构性差距：SessionStart hook 注入、命令式短 description、宽触发面积、防跳过对策、链式调用。

**约束**：
- spec-anchor 当前是 `~/.claude/skills/` 下的独立 skill，不是 Claude Code plugin
- 已有 `specanchor-boot-install.sh` 在 CLAUDE.md 中注入触发块（fallback 层）
- 已有 `spec-anchor-prelude` skill 与 superpowers 集成（但未被系统性连接）
- 必须向后兼容非插件安装方式

**目标**：
1. 将 spec-anchor 从被动 skill 升级为 plugin+skill 混合体
2. 通过 SessionStart hook 在 SpecAnchor 项目中自动注入 additionalContext
3. 重写 description 为命令式短文本
4. 内联防跳过对策到 SessionStart 注入内容
5. 与 superpowers 形成链式调用闭环

## Non-goals

- 不替代 superpowers 的工作流（brainstorm→plan→execute），而是在其之前加载 spec context
- 不在非 SpecAnchor 项目中注入任何内容（条件触发，非无条件）
- 不修改 superpowers 的代码（只通过 spec-anchor 侧的 hook 和 skill 实现集成）
- 不上架 marketplace（第一期，先验证机制有效性）
- 不修改现有 scripts/ 和 references/ 的核心逻辑

## Proposed approach

### 第一层：SessionStart Hook 注入

**1.1 创建 `package.json`**——最小化插件元数据，声明 plugin 身份，无 npm 依赖。

```json
{
  "name": "spec-anchor",
  "version": "0.7.0",
  "type": "module",
  "description": "Spec-anchored development — compile bounded, auditable engineering context for coding agents"
}
```

**1.2 创建 `hooks/hooks.json`**——注册 SessionStart hook，与 superpowers 一致的结构。

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|clear|compact",
        "hooks": [
          {
            "type": "command",
            "command": "\"${CLAUDE_PLUGIN_ROOT}/hooks/run-hook.cmd\" session-start",
            "async": false
          }
        ]
      }
    ]
  }
}
```

**1.3 创建 `hooks/session-start`**——条件注入脚本：
- 检测 cwd 是否有 `anchor.yaml` 或 `.specanchor/`
- 不是 SpecAnchor 项目 → `printf '{}\n'` 静默退出
- 是 SpecAnchor 项目 → 注入 `<IMPORTANT>` 标签的 additionalContext，包含：
  - 项目名（从 anchor.yaml 提取）
  - MANDATORY 指令（编码前必须调用 spec-anchor）
  - superpowers 链式要求（先加载 spec 再 brainstorm）
  - 5 条防跳过表（内联在注入内容中）
- 适配三平台输出格式（Cursor / Claude Code / Copilot CLI）

**1.4 创建 `hooks/run-hook.cmd`**——跨平台 polyglot 入口（cmd.exe + bash），复用 superpowers 的结构。

**关键决策**：
- 用 `<IMPORTANT>` 而非 `<EXTREMELY_IMPORTANT>` 标签——避免与 superpowers 的元技能调度器竞争优先级，两者互补不冲突
- 条件注入（检测 anchor.yaml）而非无条件注入——spec-anchor 是项目级工具，非通用技能

### 第二层：命令式短 Description

重写 SKILL.md 的 frontmatter description：

**当前**（~200 字，解释性）：
```
description: "Context Construction System for AI coding agents——为 agent 编译有边界、可审计、可沉淀的工程上下文..."
```

**目标**（~40 字，命令式）：
```yaml
description: >-
  MUST invoke before writing code in projects with anchor.yaml or .specanchor/ —
  loads coding standards, module contracts, and active tasks to anchor implementation.
```

### 第三层：宽触发面积

SessionStart hook 已覆盖项目级自动触发。Description 不再需要列举中英文关键词触发列表，只需覆盖核心场景（"before writing code"）。两层协同扩大触发面积。

### 第四层：防跳过对策

**4.1 SessionStart 注入中的内联防跳过表**（5 条）：
| "The task is too simple" → Simple tasks still must follow project conventions |
| "I already know the codebase" → Conventions change. Load current specs. |
| "I'll check specs after coding" → Late-bound spec check produces re-work |
| "Let me explore first" → spec-anchor tells you WHERE to look |
| "This is just a quick fix" → Quick fixes in wrong pattern create tech debt |

**4.2 SKILL.md 中的 `<HARD-GATE>` 区块**：
```markdown
<HARD-GATE>
This skill is MANDATORY in SpecAnchor projects. Do NOT skip boot.
Do NOT write code before the Spec Landscape is loaded.
Do NOT propose architectural decisions without referencing loaded specs.
</HARD-GATE>
```

### 第五层：链式调用

**5.1 SessionStart 注入中的链式引导**：
> "Before invoking any superpowers process skill, you MUST first invoke spec-anchor"

**5.2 spec-anchor-prelude description 优化**：
```yaml
description: >-
  MUST invoke BEFORE any superpowers process skill (brainstorming, writing-plans,
  executing-plans) in SpecAnchor projects — loads Spec Landscape first.
```

**5.3 Post-Coding Chain**（在 SKILL.md 中添加）：
编码完成后 → specanchor_check → findings → sediment proposal

### 目录结构变更

当前结构（独立 skill）：
```
spec-anchor/
├── SKILL.md          ← 根目录
├── scripts/
├── references/
└── ...
```

目标结构（plugin + skill）：
```
spec-anchor/
├── package.json      ← [新建]
├── hooks/
│   ├── hooks.json    ← [新建]
│   ├── run-hook.cmd  ← [新建]
│   └── session-start ← [新建]
├── skills/
│   └── spec-anchor/
│       └── SKILL.md  ← [移入] 从根目录移入 skills/ 子目录
├── scripts/          ← [保持]
├── references/       ← [保持]
└── ...
```

**向后兼容**：非插件安装方式仍通过 `~/.claude/skills/` 自动发现工作，只是没有 SessionStart hook。boot-install.sh 继续提供 CLAUDE.md 注入作为 fallback。

## Affected files

### 新建文件
- `package.json` — 插件元数据
- `hooks/hooks.json` — SessionStart hook 定义
- `hooks/run-hook.cmd` — 跨平台 hook 入口
- `hooks/session-start` — 条件注入脚本

### 修改文件
- `SKILL.md` → 移入 `skills/spec-anchor/SKILL.md`，重写 description + 添加 `<HARD-GATE>` + 添加 Post-Coding Chain 区块
- `references/skills/spec-anchor-prelude/SKILL.md` — 重写 description

### 不变文件
- `scripts/*` — 所有脚本保持原样
- `references/commands/*` — 所有命令定义保持原样
- `references/integrations/*` — 所有集成文档保持原样
- `anchor.yaml` — 项目配置保持原样

## Risks & open questions

| # | 风险 | 严重度 | 缓解 |
|---|------|--------|------|
| R1 | superpowers 和 spec-anchor 的 SessionStart hook 同时注入 additionalContext | 中 | Claude Code 合并多个 hook 的 context；用 `<IMPORTANT>` 而非 `<EXTREMELY_IMPORTANT>` 避免标签竞争 |
| R2 | SKILL.md 移入 `skills/` 子目录后，SKILL.md 中的相对路径引用断裂 | 高 | 所有 `references/` 路径需改为 `../../references/`；所有 `scripts/` 路径需改为 `../../scripts/`；$SA_SKILL_DIR 变量语义不变 |
| R3 | 插件化后旧 skill 安装（`~/.claude/skills/spec-anchor`）与新 plugin 安装冲突 | 中 | 文档中明确迁移步骤；plugin 安装后自动覆盖 skill 命名空间 |
| R4 | `hooks/session-start` 脚本中的 anchor.yaml 解析依赖 grep/sed | 低 | 只提取 project_name，最简解析；找不到时 fallback 到 "this project" |
| R5 | 非 SpecAnchor 项目中 hook 脚本每次 session 都运行（检测后静默退出） | 低 | 检测逻辑是两个 `-f` / `-d` 测试，开销 <1ms |

**Open Questions**：
- Q1: SKILL.md 移入 `skills/` 子目录后，`$SA_SKILL_DIR` 是指向 plugin root 还是 skill root？需确认 Claude Code 的 plugin skill discovery 机制。
- Q2: Claude Code 是否支持同时存在 `~/.claude/skills/spec-anchor/SKILL.md`（用户级 skill）和 plugin 中的 `skills/spec-anchor/SKILL.md`（plugin skill）？如果冲突如何解决？
- Q3: `<IMPORTANT>` 标签在 additionalContext 中的实际权重——是否足够让 agent 在有 `<EXTREMELY_IMPORTANT>` 的 superpowers context 存在时仍然执行 spec-anchor 的指令？
