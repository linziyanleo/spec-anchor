# Plan v1: Findings Lazy-Load (summary frontmatter + visibility-driven 分级载荷)

> 评审目标：本 plan 设计如何给 SpecAnchor 的 finding artifact 加 lazy-load 能力——frontmatter 加 `summary` 字段、写作端硬约束、bundle 按 visibility 分级载荷。
>
> 完整 Task Spec（含 frontmatter / RIPER 段落 / Acceptance Criteria）：`.specanchor/tasks/_cross-module/2026-05-25_findings-lazy-load-summary-field.spec.md`。本 v1.md 是该 Task Spec 的 review-friendly distillation，七段对应 dual-agent-review schema。
>
> 仓库：spec-anchor (CWD: /Users/fanghu/Documents/Work/spec-anchor)。

## Context / Goals

**仓库背景**：SpecAnchor 是一个为 AI coding agent 编译 bounded / auditable / sedimentable 工程上下文的 skill。v0.6 已落地 finding artifact 框架（独立文件 + frontmatter 含 visibility/freshness）和 `specanchor.context_bundle.v1` schema（含 layers.finding[]）。

**当前缺口**（grep 验证）：
- `scripts/specanchor-assemble.sh:434-445` 只做被动 `source_type` 推断（path 含 `/findings/` → 标 finding），**不主动扫描** `.specanchor/findings/` 目录。
- finding frontmatter **没有** `summary` 字段——bundle 只能给 path，agent 无法从 metadata 判断相关性。
- Bundle `layers.finding[]` 的 `load` 字段是固定标签，**不按 visibility 分级**。
- v0.6 设计文档（`findings-ledger.md` §6）声称 findings 通过 bundle 作为独立 layer，但筛选/装载策略未实装。

**目标**：
1. 在 finding frontmatter 加 `summary: <≤120 chars 单行>` 字段，写作端硬约束（template + finding.sh + validate.sh 三处）。
2. assemble 主动扫 findings 目录，按 `affects` 与 `--files` 输入做前缀匹配命中即注入 bundle。
3. 按 visibility 分三档载荷：`immediate→full` / `sediment_queue→summary` / `handoff→title-only` / `hidden→不出现`。
4. bundle `agent_instructions[]` 加 hint 防止弱模型忽略 finding。
5. 老 finding 走宽容期：candidate 强制 fail，其他 status 仅 doctor warn。

**用户价值**：agent 进入复杂任务时，bundle 默认挂出与目标文件相关的 finding 卡片（id/type/impact/affects/summary），agent 看着卡片自决是否 Read 全文，避免 bundle 体积膨胀。

## Non-goals

- 不动 `sediment proposal` 协议（summary 字段不进 SP frontmatter）。
- 不引入 inline-content bundle mode（bundle 仍然 path-only + metadata，不 inline finding 全文）。
- 不实现 `affects: contract:<name>` 类匹配（v0.6 仅支持 `module` + `path`，contract 留 v0.7）。
- 不重构 freshness 算法（继续用 mtime-based）。
- 不动 boot 脚本主流程（boot --agent 模式继续包装现有 assemble）。
- 不修改老 finding 物理文件（只在 doctor 输出 backfill 建议）。
- 不动 `specanchor.assembly.v1` 的现有字段语义（仅 `context_bundle.v1` 加新字段；旧消费者忽略未知字段不报错）。

## Proposed approach

### A. 写作端硬约束（三处协同）

**A1. finding-template.md** — frontmatter 加占位行 + 注释规则：

```yaml
---
id: F-YYYYMMDD-NNN
summary: <120 chars one-liner: 主语 + 事实 + 锚点（路径/数字/对比）>
type: ...
status: candidate
visibility: handoff
...
---
```

文末填写指引追加 "summary 字段写作准则" 段（含正反例：✓ `session TTL mismatch: doc=1h, code=24h (auth/session.go:42)` / ✗ `auth issue`）。

**A2. specanchor-finding.sh** — `new` 子命令：
- 解析 `--summary=<text>` **必选参数**
- 校验 `${#summary} -ge 1 && ${#summary} -le 120`
- 校验非占位（拒绝以 `<` 开头且 `>` 结尾的 stub）
- frontmatter 渲染时把 `summary:` 行写在 `id:` 之后
- `usage()` 文案更新（含 `--summary` Required 段）

**A3. specanchor-validate.sh** — finding lint：
- 必填字段加 `summary`
- 长度校验 ≤ 120
- 非占位校验
- `status: candidate` → fail；`status ∈ {accepted, superseded, archived}` → warn（不 fail）——给老 finding grace period

### B. 装配端主动发现 + 分级载荷

**B1. specanchor-assemble.sh** 新增 `discover_findings()` helper：
- 扫 `.specanchor/findings/*.md`
- 解析 frontmatter（如需共享 parser，抽到 `scripts/lib/`）
- 取 `affects` / `visibility` / `summary` / `id` / `type` / `impact` / `created`
- 与 `FILES_REQUESTED[]` 做前缀匹配：
  - `affects.module: <name>` → 通过 module spec 的 `module_path` 字段映射到目录前缀
  - `affects.path: <p>` → 直接前缀匹配
  - `affects.contract: <c>` → **本任务跳过**（v0.7）
- 过滤 `visibility=hidden`（不出现）
- 按 visibility 映射 `load` 标签：
  - `immediate` → `load=full`
  - `sediment_queue` → `load=summary`
  - `handoff` → `load=title`
- 排序：长前缀匹配优先 → `impact` desc（high>medium>low）→ `created` desc
- cap：默认 50（参数 `--max-findings=N`，可由 `anchor.yaml.context.budget.max_findings` 覆写）
- 注入并行数组 `FILES_TO_READ_PATHS[]` / `FILES_TO_READ_LOADS[]` / `FILES_TO_READ_REASONS[]`（保持 bash 3.2 兼容）

**B2. Bundle JSON 输出** — `print_json_bundle_v1()` 在 `layers.finding[]` 元素加新字段：

```json
{
  "id": "F-20260524-001",
  "path": ".specanchor/findings/F-20260524-001-auth-stale.md",
  "type": "contradiction",
  "impact": "high",
  "affects": ["auth"],
  "summary": "session TTL mismatch: doc=1h, code=24h",
  "load": "full",
  "freshness": "fresh",
  "freshness_reasons": [...],
  "confidence": "high",
  "source_type": "finding"
}
```

**B3. agent_instructions[]** 段追加 hint：

```text
- N findings are attached to your target files. Findings with load=full MUST be read before editing.
- For load=summary/title, scan the summary/type fields first; read full text only if relevant.
```

### C. 巡检端宽容期支持

**C1. specanchor-doctor.sh** — finding-without-summary 检查：
- 非 candidate finding 缺 `summary` → 输出 backfill warn（不 fail）
- 输出格式：`[finding] <path>: missing summary field (backfill recommended for status=<X>)`

## Risks & open questions

### 风险

- **R1（中）**：assemble 扫描成本 O(N×M)，N=findings 数 × M=输入文件数。缓解：cap=50 + 排序后截断。
- **R2（中）**：`affects` 字段是 list，单 finding 影响多模块时去重规则需 spike——同一 finding 不能在 bundle 出现两次（即便命中多个 input file）。
- **R3（高）**：强制 `--summary` 可能 break 现有调用方。已知风险点：
  - sdd-riper-one 流程中如果有自动 finding 创建路径
  - 文档示例中 `cp template ...` 手工创建模式
  - 缓解：实施前 grep 全仓 `specanchor-finding.sh new` 与 `cp.*finding-template`，确认没有自动调用方。
- **R4（中）**：validate.sh 升级在已有项目 CI 上 break。缓解：status 宽容期 + dogfood 跑现有 findings 库。
- **R5（低）**：`anchor.yaml.context.budget.max_findings` 字段是新键，需要确认 v0.6 schema 是否允许 unknown key（应该允许，因为 anchor.yaml 解析是宽容的）。
- **R6（低）**：bundle item 新字段（id/type/impact/affects/summary）对 `assembly.v1` schema 的旧消费者：旧消费者只看 `files_to_read[]` flat view，不看 `layers.finding[]`，理论零影响。需要 dogfood 一次确认。

### Open questions

- **Q1**：`affects: contract:<name>` 类匹配是否需要在本任务支持？（提议：v0.6 跳过，留 v0.7。）
- **Q2**：多个 finding 同时命中同一组目标文件时排序——`match precision desc → impact desc → created desc`（提议）or `impact desc → match precision desc → created desc`？
- **Q3**：老 finding（无 `summary` 字段）的 grace period 标识——validate 在 status=candidate 强制 fail，其他 status 仅 doctor warn（提议）；是否在某个版本之后升级为 fail？
- **Q4**：cap 默认 50 是否合理？是否需要按 visibility 分桶（immediate 不限 + 其他类共享 cap）？
- **Q5**：`load=title` 在 bundle 输出时是否仍带 `summary` 字段？（提议：是——title 指 agent 默认不主动 Read 全文，但 metadata 仍含 summary，给 agent 自决材料。）
- **Q6**：assemble 没有 `--files=` 输入时（boot 起步阶段）是否禁止扫 findings？（提议：是——避免 boot 时污染上下文，只在用户已知目标时注入。）
