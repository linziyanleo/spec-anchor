# v0.7.0 — Context Construction System + Agent Activation

**Released**: 2026-05-28

This is the first stable SpecAnchor release after the Context Construction System work. It folds the v0.6 context model and v0.7 activation work into one public release.

## Why

SpecAnchor now treats AI engineering context as four explicit layers:

- **Spec Context** — global, module, and task contracts
- **Decision Context** — checkpoint decisions with hot/cold filtering
- **Evidence Context** — verification and acceptance proof
- **Finding Context** — cross-task discoveries that can be loaded, handed off, or proposed for sediment

The goal is not to own the agent's workflow loop. The goal is to compile the right bounded context before an agent acts, expose drift after it acts, and preserve useful findings without turning every discovery into permanent spec text automatically.

## Highlights

### Context Bundle v1

`specanchor-assemble.sh --format=json --bundle-schema=context_bundle.v1` emits `specanchor.context_bundle.v1`, with explicit `layers`, `freshness`, `source_type`, `confidence`, and finding context.

`specanchor-boot.sh --agent --intent="..."` wraps the same path for agent session startup.

### Findings and sediment proposals

Findings now live as standalone `.specanchor/findings/*.md` artifacts with visibility controls:

- `immediate` findings load in full
- `sediment_queue` findings load as summaries
- `handoff` findings load as titles
- `hidden` findings stay out of the bundle

High-value findings can generate human-reviewed sediment proposals under `.specanchor/sediment/proposals/`. Proposals are never auto-applied.

### Lazy-load and summary controls

`summary` is now required for candidate findings. This keeps finding context useful under token pressure and lets Context Bundle v1 carry bounded diagnostics without changing `warnings[]` away from its compatible string-array shape.

`--max-findings=N` adds a shared cap for non-immediate finding buckets. Immediate findings remain uncapped.

### Agent activation and plugin packaging

v0.7.0 adds:

- `specanchor-boot.sh --agent`
- `scripts/specanchor-stop-triggers.sh`
- `scripts/specanchor-boot-install.sh` hook injection support
- Claude Code bootloader reference docs
- Claude Code plugin metadata and SessionStart hook support
- a plugin skill entry point at `skills/spec-anchor/SKILL.md`

Claude Code plugin install is now the recommended path when available. Skill-only installs remain supported for Cursor, Codex, Gemini, and other tools.

## Compatibility

- Existing `assembly.v1` JSON output remains the default.
- Existing `sdd-riper-one` schemas and task specs continue to work.
- Existing skill-only installs continue to work.
- Old non-candidate findings without `summary` warn instead of failing, giving existing projects a migration window.

## Not in v0.7.0

- Automatic summary backfill
- Cross-module finding affects resolution beyond the current module/path matching
- Bundle schema v3
- Automatic sediment proposal application
