# v0.5.0-beta.1 — Harness Context Control

**Released**: 2026-05-18

This release reframes SpecAnchor as a **Harness Context Control plane**. The previous tagline ("spec control plane") only covered one of three context categories that an AI engineering harness actually has to manage. v0.5.0-beta.1 makes the other two — **Decision** and **Evidence** — first-class.

## Why

LLM context is precious — long context rots in the middle, recency bias dominates, automatic compaction loses precision. SpecAnchor never tried to extend the window; it organizes what *deserves* to be in it. Until this release the only thing it organized was Spec Context (Global / Module / Task contracts).

But on every checkpoint, **~47% of human input is "add-spec"** (extend the contract along the same direction) and **~25% is "clarify"**. That's 72% of per-turn human signal. Before v0.5.0 it was thrown away as soon as the chat scrolled — agents in a new session had to re-derive it from chat history that was no longer accurate.

v0.5.0 makes both signals persistent at the right granularity:

- **Decision Context** — every checkpoint decision sediments into Task Spec §5.2 with `status` (active / superseded / withdrawn) and a `pin` flag. A lazy hot/cold view auto-prunes so the prompt only carries decisions that still matter.
- **Evidence Context** — every verification proof goes into §6.2 (Commands Run, Acceptance Criteria Mapping, Unverified Risks, Manual Checks, Rollback Handle). Acceptance criteria are auto-pinned; failed and unverified-risk entries are kept hot.

## What you get

### Schema (`sdd-riper-one` v2)

Six new sections in the Task Spec template:

| Section | Kind | Writer |
|---|---|---|
| `§1.2 Hard Boundaries` | static | author |
| `§1.3 Allowed Freedom` | static | author |
| `§4.7 Checkpoints — Contract` | static | author |
| `§5.2 Checkpoint Decisions Log` | dynamic | agent / human |
| `§6.2 Evidence Ledger` | dynamic | agent |
| `§7.2 Handoff Packet` | auto-generated | tool |

`schema.yaml` also declares a `context_control` node listing kind/writer per section. New tasks created via `specanchor_task` automatically use the v2 template.

### Configuration (`anchor.yaml`)

```yaml
context_control:
  decision_log:
    hot_window: 5
    hot_types: [redirect, rollback, halt]
    respect_phase: true
  evidence_log:
    hot_window: 5
    hot_status: [failed, unverified-risk]
    auto_pin_acceptance: true
  enforce:
    hard_boundaries: error      # error | warning | off
    allowed_freedom: warning
    checkpoints_contract: warning
    decisions_log: warning
    evidence_ledger: warning
    handoff_packet: warning
  pre_commit:
    enabled: true
    blocking: false             # warning-only by default for new projects
```

Three-layer precedence applies: CLI flags > task frontmatter `decision_log/evidence_log` > anchor.yaml `context_control.*` > builtin defaults.

### New scripts

- `scripts/lib/decision-filter.sh` — pure function lib, dual interface (`source` + CLI). Implements rules B+C from the design (status enum + lazy hot/cold).
- `scripts/lib/evidence-filter.sh` — same shape, implements rule D.
- `scripts/specanchor-doctor.sh --lint=context-control` — opt-in section presence lint. Maps `enforce` levels to existing blocking/warning machinery.
- `scripts/specanchor-assemble.sh --mode=handoff` — handoff packet renderer. With `--write-back`, refreshes §7.2 in the Task Spec.

### New command (`specanchor_handoff`)

Routes naturally from intents like *"hand this task off to a new chat"* / *"export handoff packet"* / *"换个 session 接着做"*. Definition: [`references/commands/handoff.md`](../../references/commands/handoff.md).

### pre-commit hook

`.githooks/pre-commit` now runs context-control lint after the existing `git-identity-guard`. The `pre_commit.blocking` switch decides whether a failed lint aborts the commit or only prints a warning.

> **Setup note**: SpecAnchor ships the *recipe* for the hook, not the hook file itself — consumer projects typically gitignore their local `.githooks/`. To enable, drop the snippet from this section into your repo's pre-commit hook or pipe it through `specanchor-init.sh` (planned helper). The reference implementation lives in the spec-anchor repo's local `.githooks/pre-commit` (excluded by `.git/info/exclude`); see [`docs/release/v0.5.0-beta.1.md`](v0.5.0-beta.1.md) for the inline example.

```bash
#!/usr/bin/env bash
set -euo pipefail

ROOT=$(git rev-parse --show-toplevel)
ANCHOR="$ROOT/anchor.yaml"
DOCTOR="$ROOT/scripts/specanchor-doctor.sh"

if [[ -f "$ANCHOR" ]] && [[ -f "$DOCTOR" ]] && grep -q '^  context_control:' "$ANCHOR"; then
  pc_enabled=$(awk '/^  context_control:/{cc=1;next} /^  [A-Za-z_-]+:/&&!/^  context_control:/{cc=0} cc&&/^    pre_commit:/{pc=1;next} cc&&/^    [A-Za-z_-]+:/&&!/^    pre_commit:/{pc=0} pc&&/^      enabled:/{sub(/^      enabled: */,"");sub(/[[:space:]]*#.*$/,"");print;exit}' "$ANCHOR")
  pc_blocking=$(awk '/^  context_control:/{cc=1;next} /^  [A-Za-z_-]+:/&&!/^  context_control:/{cc=0} cc&&/^    pre_commit:/{pc=1;next} cc&&/^    [A-Za-z_-]+:/&&!/^    pre_commit:/{pc=0} pc&&/^      blocking:/{sub(/^      blocking: */,"");sub(/[[:space:]]*#.*$/,"");print;exit}' "$ANCHOR")
  if [[ "$pc_enabled" == "true" ]]; then
    lint_status=0
    bash "$DOCTOR" --lint=context-control >&2 || lint_status=$?
    if [[ $lint_status -ne 0 ]]; then
      if [[ "$pc_blocking" == "true" ]]; then
        echo "[pre-commit] ❌ lint failed (pre_commit.blocking=true). Aborting." >&2
        exit $lint_status
      else
        echo "[pre-commit] ⚠️  lint reported issues (warning-only mode)." >&2
      fi
    fi
  fi
fi
```

## Migration

The lint will flag any pre-v0.5 task spec for missing sections. Two options:

1. **Recommended for active projects**: append placeholder sections marked `not applicable — legacy task` to existing task specs. Six sections, copy-paste once. The `spec-anchor` repo itself did this for its own historical tasks.
2. **Recommended for archives**: leave them, set `enforce.hard_boundaries: warning` (or `off`) in `anchor.yaml.context_control.enforce` so legacy specs don't block the doctor.

New projects start with `pre_commit.blocking: false` so the lint runs in warning-only mode until you opt in.
